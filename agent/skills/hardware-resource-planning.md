# 硬件资源规划 Skill

目标：指导 AI 基于已经接受的功能分析和器件选型，在目标板卡上规划接口、GPIO、PWM、I2C、UART、ADC 等硬件资源，确认资源占用没有冲突，并输出可用于接线、框图、代码生成和测试调试的资源分配方案。

运行时建议：本 skill 应在“器件选型”被用户接受之后使用，位于“硬件搭建”“嵌入式开发”和完整开发规划之前。它只规划当前板卡已有资源，不修改 `boards/`、`bsp/` 或 `components/`。

适用场景：

- 已经确定器件或模块，需要分配板卡接口和引脚。
- 多个模块共享 I2C、UART、PWM、GPIO，需要检查地址、通道和引脚冲突。
- 需要把器件选型转换成接线表、板级宏使用说明和固件初始化顺序。
- 需要确认当前 SDK/board 定义是否足以支持需求，或标出需要人工改板/换板的缺口。

输入：

- 用户原始需求和补充需求。
- 已接受的功能分析结果。
- 已接受的器件选型结果，包括模块、接口、供电、SDK 组件和关键参数。
- 当前 SDK 扫描摘要，包括 `components/`、`bsp/`、`boards/`、`examples/`。
- 目标板卡名称和 `boards/<board>/board.h` 中的资源宏。

核心原则：

- 必须以目标板卡已有定义为准，不要猜 GPIO。
- 优先复用 `BOARD_*` 宏、BSP 接口、driver/service 的既有资源约定。
- 允许同一 I2C 总线挂多个器件，但必须检查 I2C 地址是否冲突。
- PWM/GPIO/UART/ADC/PCNT 等资源必须逐项检查独占冲突。
- 对 bootstrap、下载、USB、Flash/PSRAM、输入仅用、输出限制等敏感引脚必须标为待确认。
- 若现有板级定义无法满足需求，只能标为缺口或建议后续新增 board 配置，不要建议直接修改 SDK 原目录来完成当前应用。
- 输出必须清楚区分“已分配”“共享可用”“冲突”“待确认”“缺口”。

规划步骤：

1. 读取器件选型
   - 提取每个器件/模块的接口类型、供电/逻辑电平、地址/通道、SDK 组件。
   - 识别必须使用的资源和可调整资源。

2. 读取目标板卡资源
   - 收集 LED、Button、Servo、I2C、UART、Motor、Encoder 等 `BOARD_*` 宏。
   - 按接口类别建立资源池：GPIO、I2C、UART、PWM/LEDC、PCNT/编码器、网络/内置外设。

3. 分配接口和引脚
   - 优先使用现有 board 宏对应的 GPIO、通道、频率和地址。
   - I2C 模块共享同一 SDA/SCL 时，列出每个器件地址。
   - PWM/舵机/电机必须列出 GPIO、LEDC timer、channel、频率。
   - UART 模块必须列出 TX/RX、波特率和是否占用调试串口。

4. 冲突检查
   - 同一 GPIO 被多个独占功能使用时标为冲突。
   - 同一 PWM channel 被多个输出使用时标为冲突。
   - 同一 UART 端口被多个模块独占时标为冲突。
   - I2C 地址相同且不可改时标为冲突；地址可改时标为待确认。
   - 未在 board 定义中出现的必需资源标为缺口或待确认。

5. 输出开发可用结果
   - 给出资源总览、详细引脚分配表、共享总线表、冲突检查表、接线建议、初始化顺序和待确认项。
   - 明确哪些 `BOARD_*` 宏和 SDK 组件会被后续代码使用。

- 目标板卡：

- 主要风险：

状态只能使用：`已分配`、`共享可用`、`冲突`、`待确认`、`缺口`。

质量检查：

- 是否覆盖器件选型中的每个器件或模块。
- 是否列出实际板级宏、GPIO、通道、地址或端口。
- 是否说明共享总线和独占资源。
- 是否显式检查 GPIO/PWM/I2C/UART 等冲突。
- 是否给出后续代码生成能直接引用的资源依据。
