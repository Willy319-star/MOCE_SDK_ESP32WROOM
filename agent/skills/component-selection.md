# 器件选型 Skill

目标：指导 AI 基于已经完成的功能分析和硬件功能清单，搜索、检索、比较并推荐合适的器件或模块，输出关键参数、接口资源、SDK 适配关系、风险和待确认项，供后续硬件资源规划、框图、代码生成、测试调试和量产设计使用。

运行时建议：本 skill 应在“功能分析”被用户接受之后使用，位于“器件选型”和“硬件资源规划”之前。它不替代硬件工程师审核，不直接输出可投产 BOM。

适用场景：

- 已有功能分析结果，需要为每个硬件功能选择具体器件或模块。
- 用户只描述了能力，如“测距”“显示”“电机驱动”“语音播报”，尚未指定型号。
- 用户给出候选器件，需要比较参数、接口、供电、SDK 支持和采购风险。
- 需要把功能清单转成可执行的模块清单、接口清单和选型依据。

输入：

- 用户原始需求和补充需求。
- 已接受的功能分析结果，包括硬件功能清单、SDK 能力映射、缺口和风险。
- 当前 SDK 扫描摘要，包括 `components/`、`bsp/`、`boards/`、`examples/`。
- 目标板卡名称和板级资源定义。
- 可用数据来源：本地 SDK 文档、示例、datasheet 摘要、用户提供的候选模块、必要时的外部搜索结果。

核心原则：

- 先从已拆分的硬件功能出发，不要凭空新增与需求无关的器件。
- 每个硬件功能至少给出一个推荐器件或模块；重要功能建议给出 2 到 3 个候选项。
- 优先选择当前 SDK 已支持或容易适配的器件/模块。
- 必须标出接口类型、供电电压、关键性能参数、板级资源占用和 SDK 适配状态。
- 如果信息来自外部搜索或 datasheet，必须保留可追溯来源；不确定的参数要标为“待确认”，不要臆造。
- 量产相关结论必须谨慎，生命周期、替代料、供应风险和认证风险都需要人工审核。
- 遵守 `prompt/prompt0.md`：不要建议修改 SDK 内部目录；如果 SDK 缺少驱动，只能标记为缺口或建议在后续版本开发组件。

选型步骤：

1. 读取功能分析结果
   - 提取每个硬件功能：输入、输出、通信、电源/结构、调试/测试。
   - 识别必须选型、可选增强、待确认和 SDK 缺口。
   - 继承功能分析中的板级资源和潜在冲突。

2. 建立候选器件检索关键词
   - 根据功能、接口、目标场景和 SDK 组件生成关键词。
   - 示例：`VL53L0X ToF I2C module`、`SSD1306 OLED 0.96 I2C`、`TB6612 motor driver ESP32 PWM`。
   - 如果用户给出型号，优先围绕该型号查关键参数和替代项。

3. 检索候选器件或模块
   - 优先使用本地 SDK 已支持的模块和 examples。
   - 再使用用户提供的 datasheet 或模块说明。
   - 如需要外部搜索，优先官方 datasheet、厂商页面、权威分销商页面；避免只依赖论坛或店铺描述。
   - 对搜索不到可靠来源的器件，标为“资料不足”。

4. 提取关键参数
   - 通用参数：型号、厂商、模块名、封装/板载模块、供电电压、逻辑电平、典型电流、峰值电流、接口类型、地址/波特率/PWM 要求、尺寸、工作温度。
   - 传感器：量程、精度、刷新率、噪声、校准需求、环境限制。
   - 执行器：驱动电压、电流、扭矩/转速/功率、控制方式、保护特性。
   - 显示/交互：分辨率、控制器、接口、显示尺寸、刷新方式、亮度/对比度。
   - 通信：协议、吞吐、延迟、天线/射频要求、认证风险。
   - 电源：输入范围、转换效率、纹波、保护、热设计。

5. 评估 SDK 和板级适配
   - 映射到当前 `driver_*`、`service_*` 或 `bsp_*` 能力。
   - 标出需要的 GPIO、I2C、UART、SPI、PWM、ADC、定时器、通道、地址。
   - 检查目标板卡资源是否足够，是否存在 GPIO、PWM channel、I2C 地址、UART 端口冲突。
   - 标出 examples 中可参考的工程。

6. 比较候选项
   - 维度：功能匹配度、SDK 适配成本、参数余量、稳定性、资料完整度、采购可得性、成本、体积、功耗、量产风险。
   - 给出推荐等级：`推荐`、`可选`、`不推荐`、`待确认`。
   - 说明推荐理由，而不是只列型号。

7. 输出待确认项和风险
   - 缺失 datasheet 或关键参数。
   - 电气不兼容，如 5V 模块接 3.3V IO。
   - 电流或功耗超出板卡供电能力。
   - I2C 地址冲突、PWM 通道冲突、UART 占用冲突。
   - 机械尺寸、安装孔、线束、连接器不确定。
   - 采购、停产、替代料、认证和量产测试风险。

推荐等级定义：

- `推荐`：功能匹配，参数足够，SDK 已支持或适配成本低，资料完整。
- `可选`：功能可用，但存在成本、参数、供货、资源或适配成本方面的权衡。
- `待确认`：关键参数、接口、电气条件、供货或结构信息不足。
- `不推荐`：明显不满足需求、电气不兼容、SDK 适配成本过高或资料风险较大。

常见功能的优先候选方向：

- LED 状态/灯效：板载 LED 或外接单色 LED，优先 `driver_led`，关注 PWM 通道、限流电阻、亮度需求。
- 按键输入：板载按键或独立轻触开关，优先 `driver_button`，关注上拉/下拉、消抖、短按/长按阈值。
- OLED 显示：SSD1306/SH1106/SSD1309 I2C OLED 模块，优先 `driver_oled`，关注 I2C 地址、分辨率、供电。
- ToF 测距：VL53L0X/同类 ToF 模块，优先 `driver_tof2000c_vl53l0x`，关注量程、视场角、环境光、I2C 地址。
- IMU 姿态：MPU6050 或同类 IMU，优先 `driver_mpu6050`，关注 I2C 地址、量程、采样率和安装方向。
- 舵机执行：SG90/MG90S/数字舵机，优先 `driver_servo`，关注供电电流、PWM 频率、扭矩和机械限位。
- 电机驱动：TB6612/DRV8833 等双路电机驱动，检查 `driver_motor`、`driver_tb6612`、`driver_encoder`、`service_pid`，关注电机电流、PWM、编码器资源。
- 语音播报：TTS UART 模块，优先 `driver_tw_tts`，关注编码格式、波特率、功放/喇叭、电源噪声。
- WiFi/上位机：优先 `service_wifi`，关注联网方式、协议、延迟、断线恢复和测试命令。

质量检查：

- 是否覆盖功能分析中的每个硬件功能。
- 是否给出了可落地的器件或模块，而不是只写能力名称。
- 是否列出关键电气、接口、性能和资源参数。
- 是否说明了 SDK 支持状态和适配成本。
- 是否标出资料来源、待确认项和人工审核点。
- 是否避免了没有来源的绝对化结论。
